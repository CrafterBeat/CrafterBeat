## What tech stacks are this week's trending projects using?

**🔍 Signal**: The strongest tech-stack signal of the week comes from the renaissance of the macOS native toolchain — the Swift + IOKit combination is now being used to do things that only system diagnostics software used to do.

[@sleepingNomad](https://news.ycombinator.com/item?id=47972511)'s **[WhatCable](https://github.com/darrylmorley/whatcable)** (527 pts / 160 comments, GitHub ⭐1,237) is the most reportable case of tech-stack deviation this week. The stack is minimal: Swift + IOKit + macOS menu bar API, zero servers, zero backend, zero subscription. IOKit is Apple's official hardware-access framework, but calling IOKit directly inside a menu bar tool to read USB-C negotiation parameters is a non-mainstream path — the vast majority of comparable tools either wrap System Information's shell, or simply invoke the `system_profiler` command-line and parse the JSON output. WhatCable skips both layers and reads kernel objects directly, earning real-time refresh, zero latency, and no permission pop-ups in return. That architectural choice led [@pimeys](https://news.ycombinator.com/item?id=47973621) to run a quick cross-platform validation: using GPT-5.5, in 10 minutes and $2, he ported WhatCable's logic into a KDE Plasma 6 QML Plasmoid — the fact that it ran proves the core logic is platform-agnostic and the IOKit calls are the only macOS lock-in point. This is the first time "AI-assisted port testing as a tech-stack evaluation method" has appeared in a top HN comment.

**[Agent-desktop](https://news.ycombinator.com/item?id=47982708)** (91 pts / 34 comments) takes a different route: a native desktop automation CLI built specifically for AI agents. The stack details aren't fully disclosed in the data, but the "CLI for AI agents" positioning is itself a tech-stack declaration — choosing CLI over GUI signals that the intended consumer is an LLM's tool call, not a human mouse. That's a direct contrast with WhatCable's menu bar positioning: both are native macOS tools, one built for humans to read, one built for agents to invoke.

**[Mljar Studio](https://news.ycombinator.com/item?id=47985077)** (60 pts / 10 comments), built by [@pplonski86](https://news.ycombinator.com/item?id=47985077), centers its stack decision on "local AI + Jupyter notebook persistence": analysis results are saved as `.ipynb` files, not a proprietary format. This is a deliberate anti-lock-in choice — trading the moat of migration costs that a proprietary data format would provide in exchange for user trust.

A clear trend emerges from this week's signals: **Swift/IOKit is becoming the default stack for macOS diagnostics tooling**, displacing the lazy path of wrapping `system_profiler` in Electron. WhatCable's ⭐1,237 and 527 pts validate the market's premium perception of "genuinely native." Agent-desktop and Mljar Studio, running alongside it, represent "CLI-first for agents" and "local inference + open format" respectively. All three projects point toward the same underlying judgment: tool developers in 2026 are actively escaping SaaS dependency, moving toward local, toward the system layer, toward open formats.

**Key judgment**: The replicable template WhatCable establishes is — Swift menu bar app + single IOKit entitlement + one-time purchase pricing (no subscription infrastructure required), build cycle measured in weeks, distribution via Mac App Store or Gumroad. The Mljar Studio template is — Python backend + Jupyter output format + local LLM inference, zero cloud costs, pricing structured around a Studio license rather than API usage billing.

**Counter view**: The IOKit path carries real platform risk. Apple has silently tightened entitlement requirements across three consecutive macOS releases. I'd want to see WhatCable's sandboxing strategy before I committed to this stack for a paid product.
